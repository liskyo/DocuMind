# 專案架構及程式碼分析

本文件詳細分析 **AI 智慧合約 (DocuMind)** 專案的目錄結構、技術架構、核心功能及程式碼邏輯。

---

## 1. 專案概述 (Project Overview)

**DocuMind** 是一個結合 AI 技術的智慧合約審閱系統。它利用大型語言模型 (Google Gemini) 來分析合約風險、重寫條款以及提供 RAG (Retrieval-Augmented Generation) 問答功能。

- **前端 (Frontend)**: 使用 React 建構，搭配 Vite 進行快速開發與構建，並使用 TailwindCSS 進行樣式設計。
- **後端 (Backend)**: 使用 Python 的 FastAPI 框架，提供 RESTful API 服務。
- **AI 核心**: 整合 Google Gemini Pro 模型進行語義分析與生成。

## 3. 技術架構分析 (Technical Stack)

### 後端 (Backend)
*   **Framework**: `FastAPI` (高效能、易於開發 API)
*   **Server**: `Uvicorn` (ASGI 伺服器)
*   **AI Model**: `Google Gemini 2.5 Pro` (透過 `google-generativeai` SDK)
*   **Document Processing**: 
    *   `pypdf`: 讀取 PDF 文件
    *   `python-docx`: 讀取與生成 Word 文件
*   **Environment**: 使用 `python-dotenv` 管理環境變數 (如 GEMINI API KEY)

### 前端 (Frontend)
*   **Framework**: `React 18`
*   **Build Tool**: `Vite` (提供極速的 HMR 開發體驗)
*   **Styling**: `TailwindCSS` (Utility-first CSS) + `Framer Motion` (動畫效果)
*   **HTTP Client**: `Axios` (與後端 API 通訊)
*   **Icons**: `Lucide React` (現代化圖示庫)

## 4. 核心功能與程式碼邏輯 (Core Logic)

### 4.1. 啟動流程 (Startup Flow)
透過根目錄的 `run.bat` 腳本，系統會執行以下檢測與啟動步驟：
1.  **環境檢查**: 確認系統是否安裝 `node` (npm) 和 `python`。
2.  **啟動後端**: 開啟新的 CMD 視窗執行 `uvicorn app:app` (Port 8000)。
3.  **啟動前端**:
    *   若首次執行，自動執行 `npm install` 安裝依賴。
    *   執行 `npm run dev` 啟動 Vite 開發伺服器。

### 4.2. API 介面 (`backend/app.py`)
主要提供以下三個核心 Endpoint：

*   `POST /api/analyze`: 
    *   接收合約檔案 (PDF/DOCX/TXT)。
    *   呼叫 `file_service.extract_text_from_file` 提取文字。
    *   呼叫 `llm_service.analyze_risk` 傳送至 Gemini 進行風險分析。
    *   回傳 JSON 格式的風險報告。

*   `POST /api/rewrite`:
    *   接收目標條款、使用者角色 (User Role)、語氣 (Tone)。
    *   呼叫 `llm_service.rewrite_clause` 生成修正後的合約條款。

*   `POST /api/rag`:
    *   接收使用者問題 (Query) 與合約片段 (Context)。
    *   呼叫 `llm_service.rag_query` 讓 AI 基於合約內容回答問題。

### 4.3. AI 服務層 (`backend/services/llm_service.py`)
這是系統的大腦，封裝了與 Gemini API 的互動：
*   **API Key 管理**: 支援多組 API Key 輪詢 (`get_random_key`) 以分散請求配額。
*   **Prompts**: 引入 `prompts` 資料夾內的定義好的 String Templates，確保模型輸出的格式穩定 (主要要求輸出 JSON)。
*   **Error Handling**: 包含基本的錯誤捕獲，防止 API 呼叫失敗導致 Server Crash。

### 4.4. 前端互動 (`frontend/src/App.jsx`)
*   **狀態管理**: 使用 `useState` 管理 `analyzing` (分析中), `result` (分析結果), `lang` (語言設定)。
*   **檔案上傳**: 使用 `<input type="file" />` 接收檔案並透過 `axios` 送往後端。
*   **UI 組件**:
    *   `RiskMonitor`: 展示風險等級、問題描述與修正建議。
    *   `QABot`: 側邊欄聊天室，提供針對合約內容的問答。
*   **多語言**: 支援動態切換 中/英 介面 (`translations.js`)。

## 5. 觀察與建議 (Observations & Recommendations)
1.  **安全性**: API Key 目前似乎是透過環境變數或直接寫在某些設定中讀取，建議確保 `.env` 檔案不被納入版本控制 (已在 `.gitignore` 中見到)。
2.  **穩定性**: `run.bat` 依賴 User 本機的 Python/Node 環境。若移至其他機器，建議使用 Docker 容器化部署以確保環境一致。
3.  **模型版本**: 目前程式碼指定 `gemini-2.5-pro`，需確認該模型版本名稱是否正確對應 Google API 的最新發布名稱 (通常是 `gemini-1.5-pro` 或 `gemini-pro`)，以免呼叫失敗。
